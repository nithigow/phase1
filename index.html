Phase 1 — Input Ingestion: Production Workflow
Processing: Azure Storage · PostgreSQL · Health/Auth Verification · RAG
Step 1 — User fills intake form
Project, URL, Files (BRD/FWB/WSB), Assumptions, Creds, Endpoints
Step 2 — Initialize PostgreSQL Record
Create project_id · Store metadata & assumptions
PostgreSQL stores: name, app_url, user assumptions, and raw API endpoint list.
Step 3 — URL & Credential Verification
Ping URL · Functional login check with provided creds
Verify App URL is live (2xx)
Attempt Auth handshake with Creds/Endpoints
If fail → Stop ingestion & notify user
Step 4 — Upload to Azure Blob Storage
Store .pdf and .docx files permanently
Upload files to Azure Bucket.
Save Azure file URLs into PostgreSQL 'Files' table.
Step 5 — Extract text from documents
Direct text parsing (No OCR required)
PDF → PyMuPDF
DOCX → python-docx
Step 6 — Chunk text & assumptions
Split content into semantic sections
Combines document content and manually provided assumptions for better RAG context.
Step 7 — Generate embeddings
Convert chunks into vector representations
Step 8 — Store in Vector Database (pgvector)
Link vectors to project_id in PostgreSQL
Relational table links the vector to the project metadata for structured retrieval.

Step 1 & 2 Intake & PostgreSQL Initialization

The system captures the expanded intake data and immediately persists it into a PostgreSQL relational database.

  • Project Metadata — Name, URL, and User-defined Assumptions.
  • Credentials — Encrypted storage of roles and passwords.
  • API Endpoints — Target endpoints to be tested against.
Step 3 Health & Auth Verification

Critical validation step to ensure the testing environment is ready before processing knowledge.

  • System verifies application reachability.
  • System performs a mock login or auth-handshake using provided credentials.
  • Failure at this stage prevents unnecessary resource consumption (API/Tokens).
Step 4 & 5 Azure Storage & Extraction

Documents are stored for cross-phase accessibility and audited text extraction.

  • Azure Blob Storage — Acts as the permanent repository for BRD/FWB/WSB files.
  • Simplified Extraction — Per teacher's guidance, system uses direct text parsing (PDF/DOCX) only.

Step 6, 7 & 8 RAG Knowledge Pipeline

Text is prepared for the LLM agents in Phase 2 using the RAG architecture.

  • Chunking — Breaks document text and assumptions into searchable fragments.
  • Embeddings — Numerical representation of semantic meaning.
  • pgvector Integration — Embeddings are stored in PostgreSQL, allowing for relational filtering (e.g., retrieving only chunks belonging to a specific Project ID).
