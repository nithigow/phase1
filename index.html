Step 1 User fills intake form

This is the entry point of the system. The user opens the web UI or calls the ingestion API and submits the required information to start a testing project.

  • Project name — e.g. "Health Insurance Portal".
  • App URL — target application URL.
  • Environment — staging / dev / production.
  • Files — BRD, FSD or supporting documents.
  • Credentials — login credentials for each role.
  • API endpoints — Swagger URL or manual list.
Example — credentials
Role: admin | Email: admin@test.com | Password: Admin@123
Role: user | Email: user@test.com | Password: User@123

The backend receives this as a single request and starts the ingestion pipeline.

Step 2 Validate inputs

The system validates the request before any processing begins.

  • Project name must exist
  • Application URL must be valid
  • At least one BRD document must be uploaded
  • Allowed formats: .pdf or .csv
  • At least one credential set must be provided

If validation fails the request is rejected and the ingestion pipeline stops immediately.

Step 3 Create project record

A project entry is created in the database. This record becomes the reference point for the entire testing lifecycle.

Example project record
project_id: a3f2c1d4...
project_name: Health Insurance Portal
environment: staging
status: ingestion_in_progress

The system uses this project ID to associate documents, embeddings and test results.

Step 4 URL health check

The system verifies that the application is reachable before processing documents.

  • Send HTTP GET request to application URL
  • Timeout after 10 seconds
  • If response status ≥ 400 → ingestion stops
Example
If the staging server is down the system returns:
"Application not reachable. Please verify the URL."
Step 5 Process uploaded files (in memory)

Uploaded documents are processed directly in memory. No raw files are written to disk or external storage.

  • Files are read from the request stream
  • Stored temporarily in memory buffers
  • Immediately passed to the extraction pipeline
Example
User uploads brd.pdf → FastAPI reads file → stored in memory → sent to extraction.
Step 6 Detect file type

The system determines how the document should be processed.

  • PDF documents → processed with PyMuPDF
  • CSV files → parsed using pandas
  • Scanned PDFs → routed to Vision LLM OCR
Detection rule
If extracted text length is extremely small on a multi-page PDF, the document is classified as scanned.
Step 7 Extract raw text

The system extracts textual content from the uploaded documents.

  • Text PDFs → PyMuPDF extraction
  • Scanned PDFs → OCR using Vision LLM
  • CSV → converted to text format
Output
All extraction paths produce a single plain text string representing the document content.
Step 8 Chunk document text

The extracted document text is divided into smaller chunks before embedding generation.

  • Typical chunk size: 300–800 tokens
  • Overlap between chunks preserves context
  • Improves semantic search accuracy
Example chunks
Chunk 1 → "User must be above 18 years old"
Chunk 2 → "Password must contain at least 8 characters"
Step 9 Generate embeddings

Each chunk is converted into a vector embedding using an embedding model.

  • Embeddings represent semantic meaning
  • Used for similarity search in RAG
  • Forms the knowledge base for requirement retrieval
Example embedding
chunk_text: "User must be above 18 years old"
embedding: [0.012, -0.441, 0.332, ...]
Step 10 Store embeddings in vector database

The generated embeddings are stored in a vector database such as pgvector.

  • Each chunk is stored with its embedding
  • Allows semantic search against BRD content
  • Supports the RAG architecture used in Phase 2
Vector DB record
project_id: a3f2c1d4...
chunk_text: "User must be above 18 years old"
embedding: vector(1536)
Step 11 Update project status

Once ingestion and embedding generation are completed, the project status is updated.

  • Status changes to ingestion_complete
  • This signals Phase 2 agents to begin processing
  • Vector database now contains BRD knowledge
Final system state
projects table → status = ingestion_complete
vector database → embeddings stored
credentials table → encrypted login credentials

Phase 2 agents will now use RAG to retrieve BRD context and generate structured requirements and test cases.
